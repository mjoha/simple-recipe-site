# Feature: Restructure Output Folder And Split Source From Generated Assets

## Goal

Replace the legacy `wwwroot/` folder name and stop mixing hand-authored source assets with generated output in the same directory.

The current layout is a leftover from the original ASP.NET Core baseline:

```text
wwwroot/
  index.html      # generated by scripts/build-site.mjs
  styles/site.css # hand-authored source
  scripts/search.js # hand-authored source
```

This is confusing for future agents and humans because:

- `wwwroot/` is ASP.NET-specific naming and we no longer have a .NET runtime.
- Generated and source files live side by side, so it is not obvious which files are safe to edit by hand.
- A few remaining identifiers in scripts, workflow, package config, and active docs still refer to `wwwroot` and `recipe`, even though the project is a generic static Markdown catalog generator.

This feature reorganizes the layout, renames artifacts where the old name leaks, and updates docs to match. It is structural and naming only.

---

## Scope

### Include

- Introduce a `src/` folder for hand-authored browser assets.
- Introduce a `dist/` folder as the generated output target.
- Update `scripts/build-site.mjs` to write generated HTML to `dist/` and copy source assets from `src/` into `dist/`.
- Rename `scripts/serve-wwwroot.mjs` to a generic name and serve `dist/`.
- Update `scripts/verify-local.sh` to check `dist/` paths and use a non-recipe log filename.
- Update `package.json` scripts to reference the new server script.
- Update `.github/workflows/deploy-pages.yml` to commit and upload from `dist/`.
- Update `.gitignore` if needed so the new layout works on a clean clone.
- Remove remaining `wwwroot` and `recipe` references from active (non-archive) files.
- Update active `.ai` docs and `README.md` to match.

### Exclude

- No changes to generated visual output, layout, typography, or behavior.
- No changes to content authoring (`content/index.md`, `content/items/*.md`).
- No backend, database, or TypeScript reintroduction.
- No new dependencies.
- No changes to archived feature specs in `.ai/archive/features/`. Their historical wording stays as-is.

---

## Target Folder Structure

```text
content/
  index.md
  items/
    *.md

src/
  styles/
    site.css
  scripts/
    search.js

dist/
  index.html       # generated
  styles/
    site.css       # copied from src/styles/site.css at build time
  scripts/
    search.js      # copied from src/scripts/search.js at build time

scripts/
  build-site.mjs
  serve-dist.mjs   # renamed from serve-wwwroot.mjs
  verify-local.sh
```

`dist/` is the deployable artifact. Anything under `dist/` is reproducible from `content/` plus `src/` plus `scripts/build-site.mjs`.

`src/` is the only place where browser-facing CSS and JS are edited by hand.

---

## Build Script Changes

`scripts/build-site.mjs` should:

- Read `content/index.md` and `content/items/*.md` as today.
- Generate `dist/index.html` (was `wwwroot/index.html`).
- Copy `src/styles/site.css` to `dist/styles/site.css`.
- Copy `src/scripts/search.js` to `dist/scripts/search.js`.
- Create `dist/`, `dist/styles/`, and `dist/scripts/` if they do not exist.

The generated HTML should keep referencing `./styles/site.css` and `./scripts/search.js` so relative paths continue to work for GitHub Pages project sites.

The build script should be idempotent and safe to re-run.

If a simple full-folder approach is easier (delete `dist/` then regenerate), that is acceptable, as long as it is fast and does not delete files outside `dist/`.

---

## Dev Server Changes

Rename `scripts/serve-wwwroot.mjs` to `scripts/serve-dist.mjs` and update it to serve from `dist/`.

Behavior should be unchanged otherwise:

- Listens on `127.0.0.1` at the same port (default `5002`, overridable via `PORT`).
- Serves `index.html` for `/`.
- Refuses to escape the served directory.
- Same content-type map.

Update `package.json`:

```json
{
  "scripts": {
    "build": "node scripts/build-site.mjs",
    "dev": "npm run build && node scripts/serve-dist.mjs",
    "verify": "./scripts/verify-local.sh"
  }
}
```

---

## Verify Script Changes

Update `scripts/verify-local.sh`:

- Replace the `wwwroot/...` path checks with `dist/...` equivalents.
- Replace the log filename `simple-recipe-site-verify.log` with a generic name such as `static-markdown-catalog-verify.log`.
- Drift check should diff `dist/index.html` (and any other tracked generated files in `dist/`).
- Asset existence checks should target `dist/styles/site.css` and `dist/scripts/search.js`.
- Content checks (search input id, details elements, A-Z markers, no `id="status"`) should run against `dist/index.html`.

If `dist/` is gitignored (see below), the drift check should still work for any tracked generated files. If `dist/index.html` is no longer tracked, replace the drift check with a structural check that the file exists and contains expected markup after `npm run build`.

---

## GitHub Pages Workflow Changes

Update `.github/workflows/deploy-pages.yml`:

- After `npm run build`, the upload artifact path should be `dist`.
- The auto-commit step should commit `dist/index.html` (and any other tracked generated files) if they changed, or be removed entirely if `dist/` is gitignored.
- No references to `wwwroot/`.
- No references to recipe JSON.

Pick one of two options for committing generated output:

- Option A: keep committing generated output to `main` (current behavior). Track `dist/index.html`. Workflow auto-commits on regenerate.
- Option B: stop committing generated output. Add `dist/` to `.gitignore`. Workflow builds fresh on every deploy.

Recommended: Option B, because it removes the auto-commit-to-main step, removes drift entirely, and matches typical static-site conventions. Document the choice in `README.md`.

If Option B is chosen:

- Add `dist/` to `.gitignore`.
- Remove the auto-commit step from the workflow.
- Update verify script to skip the git diff drift check and rely on structural checks.

## Verification

After implementation, verify:

- `npm run build` succeeds and writes `dist/index.html`, `dist/styles/site.css`, `dist/scripts/search.js`.
- `dist/index.html` references `./styles/site.css` and `./scripts/search.js`.
- `npm run dev` serves the site from `dist/` at `http://localhost:5002`.
- `npm run verify` succeeds.
- `wwwroot/` no longer exists in the working tree.
- Searching for `wwwroot` in active files returns no results, except inside `.ai/archive/features/` where historical wording is preserved.
- Searching for `recipe` (case-insensitive) in active files returns matches only in legitimate places: example content under `content/items/`, the catalog title in `content/index.md` if the user keeps it, and any archived feature specs.
- The site rendered from `dist/` looks and behaves the same as before this feature.

## Acceptance Criteria

- Source assets live under `src/` and are not mixed with generated output.
- Generated output lives under `dist/`.
- The legacy `wwwroot/` folder is gone.
- The legacy `serve-wwwroot.mjs` script is renamed.
- Build, dev, verify, and GitHub Pages deploy all work end-to-end against the new layout.
- Active docs and configuration no longer reference `wwwroot` or use recipe-specific framing as if it were the product.
- No visual or behavioral changes to the generated site.
- Archived feature specs are unchanged.
